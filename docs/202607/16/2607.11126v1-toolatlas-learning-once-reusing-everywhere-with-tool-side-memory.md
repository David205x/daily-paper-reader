## 摘要
大型语言模型（LLM）智能体日益依赖由共享提供商提供、并被异构下游智能体访问的外部工具。现有方法通过参数更新、提示优化或智能体侧记忆来改进工具使用，导致工具知识难以共享，且仅限于过去任务中观察到的行为。我们认为，可复用的工具知识应由工具提供商维护。我们提出ToolAtlas，一个基于图的框架，通过执行验证的探针构建工具能力、失败边界和跨工具组合的持久化提供商侧工具记忆。在推理时，智能体通过自适应图遍历查询工具记忆。在两个基于MCP的基准测试（涵盖八项服务）上，ToolAtlas在pass@1和pass@4上分别比现有工具侧优化和智能体侧记忆基线高出最多21.61%和18.61%。相同的工具记忆还能跨环境实例和智能体框架迁移，无需重新训练或任务时探索，在pass@1/pass@4上分别获得最多24.16%/16.22%和17.49%/14.27%的相对提升。消融研究表明，这些提升源于将工具中心化的记忆组织与能力引导的执行探针相结合。这些结果确立了提供商侧工具记忆作为工具服务器有效且可复用的范式。我们的代码位于：https://github.com/PuppyKnightUniversity/ToolAtlas。

## Abstract
Large language model (LLM) agents increasingly rely on external tools served by shared providers and accessed by heterogeneous downstream agents. Existing approaches improve tool use on the agent side through parameter updates, prompt refinement, or agent-side memory, making tool knowledge difficult to share and limited to behaviors observed in past tasks. We argue that reusable tool knowledge should instead be maintained by the tool provider. We introduce ToolAtlas, a graph-based framework that builds a persistent provider-side tool memory of tool capabilities, failure boundaries, and cross-tool compositions through execution-verified probing. At inference time, agents query the tool memory via adaptive graph traversal. Across two MCP-based benchmarks spanning eight services, ToolAtlas outperforms existing tool-side optimization and agent-side memory baselines by up to 21.61% in pass@1 and 18.61% in pass@4. The same tool memory also transfers across environment instances and agent frameworks without retraining or task-time exploration, yielding up to 24.16%/16.22% and 17.49%/14.27% relative gains in pass@1/pass@4, respectively. Ablation studies show that these gains arise from combining tool-centered memory organization with capability-guided execution probing. These results establish provider-side tool memory as an effective and reusable paradigm for tool servers. Our code is in: https://github.com/PuppyKnightUniversity/ToolAtlas.

### 1. 核心问题与整体含义

-   **研究动机**：大语言模型（LLM）智能体越来越依赖外部工具（如文件系统、GitHub API等）。现有方法通常在智能体侧优化工具使用（如微调模型参数、优化提示词、或构建智能体自身的“记忆”）。这导致两个关键问题：
    -   **泛化困境**：每个智能体学到的工具知识被“困在”该智能体内，无法与其他使用相同工具的异构智能体共享。同一工具行为需被不同智能体重复“重新发现”。
    -   **能力盲目**：智能体侧记忆仅记录过去任务中观察到的工具行为，遵循任务分布而非工具本身的能力边界。这使得工具的未探索能力、失败边界和跨工具组合模式被遗漏。
-   **核心主张**：可复用的工具知识不应驻留在智能体侧，而应由**工具提供商**来维护。这种“提供商侧记忆”可以一次性构建，被所有下游智能体复用，从而解决上述问题。

### 2. 方法论：核心思想与技术细节

-   **核心思想**：ToolAtlas 为每个 MCP（Model Context Protocol）服务器构建一个持久化的“工具记忆图”。这个图记录工具的能力、失败边界和组合方式，并通过**能力引导的探索**（而非被动记录）来主动扩展知识边界。在推理时，智能体通过**自适应图遍历**来查询这个记忆图，获取任务相关的指导。

-   **关键技术细节**：
    1.  **三层的工具记忆图 (MT)**：
        -   **工具-追踪图 (Gtrace)**：将过去任务执行的原始轨迹（通常带有智能体特有的格式和失败尝试）蒸馏成**智能体中立的追踪**（即一系列 (工具, 理由) 对）和任务级提示。节点是追踪节点，边是语义相似度。
        -   **工具-能力图 (Gcap)**：以工具为中心。每个工具节点包含三组条目：
            -   **能力条目 (Ui)**：该工具可靠支持的、已验证的有效用途。
            -   **边界条目 (Bi)**：该工具会变得不可靠或不可行的场景。
            -   **组合条目 (Ci)**：与其他工具反复组合使用的模式。
        -   **工具-策略图 (Gstrat)**：从多个任务中提取的高阶编排原则（策略节点），例如“在操作前先确定工作空间范围”。

    2.  **能力探索 (Capability Exploration)**：
        -   **机制**：与被动记录任务轨迹不同，ToolAtlas 使用一种**前沿探索**的迭代循环。它利用当前工具-能力图的“已知”状态作为反馈，指导LLM生成测试任务，主动探索**未被记录的**能力（向内填补）和**潜在失败边界**（向外推展）。
        -   **流程**：从前沿探索中提出的新任务 → 由智能体执行 → 使用验证器检查结果 → 将执行验证后的结果通过“记忆归纳”更新回三层图中。

    3.  **推理时的自适应内存遍历**。
        -   **机制**：针对新任务 `Q`，检索最相似的k个追踪节点作为初始上下文。然后，一个轻量级导航器动态执行图遍历操作（如读取追踪、扩展相似节点、读取工具能力、读取策略），直到收集到足够信息。
        -   **输出**：将收集到的信息压缩成一个**任务条件化的指导**（guidance），包含“候选种子”、“操作手册”、“策略”和“工具提示”，注入到用户智能体的提示中。

-   **算法流程**：
    -   **阶段1：记忆引导**：为每个工具生成种子任务，执行并验证轨迹，通过记忆归纳构建初始图。
    -   **阶段2：能力探索**：基于当前图，采样子图，提出探索新能力和边界的新任务，执行并更新图。重复多轮。
    -   **阶段3：推理**：接收新任务，进行自适应图遍历，生成任务指导，辅助下游智能体完成实际任务。

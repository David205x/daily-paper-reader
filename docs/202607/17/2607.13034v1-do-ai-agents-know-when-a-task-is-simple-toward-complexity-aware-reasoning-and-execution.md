## 摘要
大型语言模型（LLM）智能体日益自动化多步骤工程与信息学工作流，但它们很少询问任务实际需要多少努力。它们常遵循“最大上下文优先”策略——重新读取已经看过的文件和依赖——将一行编辑变成小型代码库审计。我们认为缺失的能力是任务感知的执行范围估计：在投入预算前判断任务难度、真正需要的信息以及最短可靠路径。我们形式化了最小充分执行和智能体认知冗余比（ACRR），并提出E3（估计、执行、扩展）：智能体估计初始操作点，执行最小可行路径，仅在验证失败时扩展范围。在MSE-Bench（一个能力控制模拟器中包含121次编辑的确定性基准）上，E3以100%的成功率匹配最强基线，同时削减成本85%、令牌数91%和检查文件数92%，并进一步以16%的优势击败强自适应检索基线；这些收益在保留指令措辞和几乎所有成本加权下依然成立。一个配套的真实模型框架（LLM-Case）在实时gpt-4o智能体编辑真实开源库时证实了效果，每个候选补丁通过实际运行项目的真实pytest套件并对照测量预言机进行评分：过度读取较温和但真实存在，E3在可比任务成功率下是最精简且最快的策略——其唯一缺陷是提供商速率限制，而非错误编辑。我们将此视为执行冗余的控制性探针，而非对任何已部署智能体的测量，并将任务感知执行定位为迈向工程化AI（EGAI）的一步——智能体的努力与任务的工程现实紧密相连。我们发布了该框架和基准。

## Abstract
Large language model (LLM) agents increasingly automate multi-step engineering and informatics workflows, yet they rarely ask how much effort a task actually requires. They often follow a maximum-context-first strategy--re-reading files and dependencies they have already seen--turning a one-line edit into a small code-base audit. We argue the missing capability is task-aware execution-scope estimation: judging a task's difficulty, the information it truly needs, and the shortest reliable path before committing budget. We formalize minimum-sufficient execution and the Agent Cognitive Redundancy Ratio (ACRR), and propose E3 (Estimate, Execute, Expand): the agent estimates an initial operating point, executes a minimum viable path, and expands scope only when verification fails. On MSE-Bench--a deterministic benchmark of 121 edits in a capability-controlled simulator--E3 matches the strongest baseline's 100% success while cutting cost by 85%, tokens by 91%, and inspected files by 92%, and further beats a strong adaptive retrieval baseline by 16%; the gains survive held-out instruction wording and essentially every cost weighting. A companion real-model harness (LLM-Case) corroborates the effect on a live gpt-4o agent editing a real open-source library, with every candidate patch graded by actually running the project's real pytest suite against a measured oracle: the over-reading is milder but real, and E3 is the leanest and fastest policy at comparable task success--its one shortfall a provider rate-limit, not a wrong edit. We frame this as a controlled probe of execution redundancy, not a measurement of any deployed agent, and position task-aware execution as a step toward engineering-grounded AI (EGAI)--agents whose effort is anchored in the engineering reality of the task. We release the framework and benchmark.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：大语言模型（LLM）驱动的自主智能体在执行任务时，常常不区分任务难度，默认采用“最大上下文优先”策略——重新读取已经看过的文件与依赖，把一行简单的编辑变成一次小型代码库审计。这种过度资源消耗行为源于智能体缺乏**任务复杂度感知**能力：无法快速判断任务有多难、真正需要哪些信息、以及最短可靠路径。
- **背景与动机**：随着LLM智能体在软件工程、数据分析和科学计算中的广泛应用，效率与正确性同样重要。作者观察到，即使是简单的关键词替换（如个人网站图标更换），一个强大智能体也可能花费数分钟重读整个项目。这种冗余不是知识或检索失败，而是缺失一种元认知判断：**先判断任务难度，再行动**。这呼应了人类工程师的行为——快速估计难度和范围，从最小计划开始，失败再扩大搜索。
- **整体含义**：真正高效的智能不仅在于能解决难题，更在于能**识别何时任务是简单的**并相应行动。论文将这种能力形式化为“任务感知执行范围估计”，并提出E3框架，旨在使智能体的努力与工程现实相匹配，作为“工程化AI”（EGAI）的一步。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：E3框架替代了“最大上下文优先”的循环，采用三阶段过程：**估计（Estimate）→ 执行（Execute）→ 扩展（Expand）**。
  - **估计阶段**：智能体从查询`q`、环境`E`和先验经验`M`中构建一个**初始操作点**`x0 = (ˆd, ˆs, ˆr, ˆc)`，其中`ˆd`为估计难度，`ˆs`为估计范围（需触及的文件或站点），`ˆr`为风险，`ˆc`为置信度。该估计是廉价且乐观的，基于词汇线索和一次廉价结构探针。
  - **执行阶段**：根据`x0`，智能体执行**最小可行路径**：范围级别`ℓ`决定定位、编辑、验证的粒度。例如，级别1仅单文件定位并编辑；级别2复用缓存搜索结果并编辑直接站点；级别3额外跟踪导入（`dependency_trace`）以触及间接站点。验证努力与风险成比例。
  - **扩展阶段**：若验证失败，不从头开始，而是逐级增加范围（`ℓ ← ℓ+1`），重用已学信息（缓存搜索命中），并重新规划。扩展有最大次数`K`的约束，确保在最坏情况下降级为穷举策略，但在常见情况下保持精简。

- **关键技术细节与公式**：
  - **轨迹成本**：`C(π) = α·T_lat + β·N_tok + γ·N_tool + δ·N_file`，其中`T_lat`为墙钟延迟，`N_tok`为推理和上下文令牌数，`N_tool`为工具调用次数，`N_file`为完全拉入上下文的独特文件数。权重可配置，默认设置中文件权重`δ`最大。
  - **最小充分执行**：定义最小成本轨迹满足可靠性目标`1-ε`：`π⋆ = argmin_π C(π) s.t. P(success|π,τ) ≥ 1-ε`。其成本`C_min(τ)`是任务应需的努力。
  - **智能体认知冗余比（ACRR）**：`ACRR(τ) = (C_act(τ) - C_min(τ)) / C_min(τ)`，仅对成功运行定义。ACRR=0表示最优精简，ACRR=4表示花费了5倍必要成本（400%冗余）。
  - **初始操作点**：`x0 = f(q, E, M)`，由查询`q`、廉价环境探针和先验经验估计得出，无需精确答案，但能指导后续搜索快速稳定。

- **算法流程**（文字描述）：
  - 输入：任务`τ = (q, E, V)`，估计器`f`，最大扩展次数`K`。
  - 1. 通过`f`估计初始操作点`x0`（最多一次廉价探针）。
  - 2. 根据估计难度`ˆd`设置初始范围级别`ℓ`；缓存搜索命中。
  - 3. 以当前`ℓ`、`x0`和缓存执行最小可行路径，尝试完成并验证。
  - 4. 若验证失败且`ℓ<3`且扩展次数`k<K`，则增加`ℓ`，重新执行。
  - 5. 循环最多K次，返回最终成功状态。

## 6. 论文的主要结论与发现

1. **当前智能体存在显著冗余**：最大上下文优先策略在简单任务上ACRR高达12.9（花了13倍必要努力），且冗余程度随任务变简单而增长（A级1级ACRR 22.1，3级5.4）。
2. **E3可同时实现高成功率和低成本**：在MSE-Bench上，E3以100%成功率匹配最强基线（MCF和AR），同时**成本降低85%、令牌数降低91%、检查文件数降低92%**；相对强自适应AR基线，成本仍降低16%。
3. **估计和扩展相互协作**：估计降低成本，扩展保证可靠性。移除扩展导致成功率下降，移除估计则成本上升。
4. **鲁棒性**：E3在保留指令措辞和几乎任意成本权重下仍是成本最低的完全成功策略；若估计器不准确，扩展阶段仍能挽回。
5. **真实模型验证**：在gpt-4o上，E3是总体最精简且最快的策略。过读取现象在真实模型上较温和但真实存在，且过度读取会因速率限制或预算耗尽导致任务失败。
6. **工程直觉**：通过电力系统潮流牛顿-拉夫逊初值点类比，说明了**良好的初始操作点能显著减少迭代次数并保持收敛可靠**，支撑E3设计。

## 8. 不足与局限：包括实验覆盖、偏差风险、应用限制等

- **模拟器局限性**：主要实验在能力控制模拟器中进行，不涉及真实语言模型，因此数值是成本模型下的特性，不能直接迁移到任何特定商业智能体。成本模型虽经权重扫描，但现实中令牌、延迟、文件读取的相对代价可能不同。
- **基准范围有限**：MSE-Bench仅覆盖文件级编辑（直接站点/间接站点通过导入依赖），未包括动态调度、运行时反射、配置耦合等其他隐藏复杂度源泉。任务从少数原型（每级一种）程序生成，并非开放世界的多样任务。
- **真实模型验证规模小**：LLM-Case仅运行了5个任务各3次，共15次运行，且仅使用gpt-4o，结果作为案例研究而非有统计功效的基准。速率限制、模型行为随机性可能导致结论不稳定（如在L3任务上，哪一策略失败在不同轮次间波动）。
- **估计器设计简单**：当前估计器是基于词汇规则和一次探针的透明方法，在任务措辞变化时准确率下降（从85.1%降至66.9%）。学习型或LLM支持的估计器可能泛化更好，但未实现和测试。
- **未考虑模型编辑能力与范围的交互**：模拟器中假设只要看到相关内容就能正确编辑，但真实模型中，更广泛上下文可能提高编辑准确率（或避免错误）。当前设计未能捕获模型能力-范围权衡，即更广上下文可能对复杂任务更安全，但论文未量化这类增益。
- **应用限制**：实验聚焦于代码编辑场景，结论向其他领域（如数据流水线、对话系统）推广需谨慎。此外，E3假设验证器（V）存在且可靠，但在需要人类判断的任务中可能不易获得。

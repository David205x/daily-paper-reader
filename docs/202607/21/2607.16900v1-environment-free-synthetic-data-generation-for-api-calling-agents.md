## 摘要
训练API调用的大语言模型（LLM）代理需要大量高质量轨迹。然而，大规模收集此类数据通常需要完全实现的环境，包括可执行的API和预填充的逼真后端数据库，这成为可扩展性的主要瓶颈。为解决这一问题，我们提出了一种无环境合成数据生成方法，利用LLM作为即时数字世界模型。仅给定API规范，我们的方法生成模拟代理与有状态环境交互的轨迹。具体而言，LLM首先生成可使用给定API解决的多样化任务。然后，教师代理迭代解决每个任务，同时LLM模拟器根据任务上下文和模拟历史生成连贯的合成API响应。最后，LLM评审员筛选轨迹以确保数据集质量。我们在具有挑战性的AppWorld和OfficeBench基准上评估了我们的方法，这些基准包括信息检索和状态更改任务。在我们的合成数据上微调模型带来了显著的性能提升，表明无需任何可执行环境即可为API调用代理生成有效监督。我们的结果证明了基于LLM的API模拟是跨多样API生态系统训练代理的一种实用、可扩展的解决方案。

## Abstract
Training API-calling large language model (LLM) agents demands massive amounts of high-quality trajectories. However, collecting such data at scale typically requires fully implemented environments with executable APIs and realistic, pre-populated backend databases, creating a major bottleneck for scalability. To overcome this, we propose an environment-free synthetic data generation approach that leverages LLMs as on-the-fly digital world models. Given only API specifications, our method generates trajectories mimicking interactions between an agent and a stateful environment. Specifically, an LLM first generates diverse tasks solvable with the provided APIs. A teacher agent then iteratively solves each task while an LLM simulator generates coherent synthetic API responses conditioned on the task context and simulation history. Finally, an LLM judge filters the trajectories to ensure the quality of the resulting dataset. We evaluate our approach on the challenging AppWorld and OfficeBench benchmarks, which include both information-retrieval and state-changing tasks. Fine-tuning models on our synthetic data yields significant performance gains, demonstrating that effective supervision for API-calling agents can be generated without any executable environment. Our results establish LLM-based API simulation as a practical, scalable solution for training agents across diverse API ecosystems.

- **整体贡献**：提出**ESAT**（Environment-Free Synthetic Agentic Trajectory generation）流水线，仅需API规范即可生成多步交互轨迹，完全绕过环境建设，实现可扩展的代理训练数据合成。

## 2. 方法论
### 核心思想
- 使用LLM模拟有状态的API环境（称为“API模拟器”），与教师代理交互生成轨迹。
- 整个流水线分为三个独立阶段，无需任何真实API执行。

### 关键技术细节
#### 阶段1：任务合成（Task Synthesis）
- **分桶生成（Bucketized Generation）**：按难度（easy/medium/hard）、动作类型（read/write/mixed）、任务焦点（constraint satisfaction/derivation/iteration/open）、涉及app数（1-3）、每app API数范围组合成360个桶，确保覆盖。
- **逆频率采样**：根据已生成任务中app和API的使用频率，优先选择未充分使用的app/API，保证多样性。
- **任务过滤与重写**：LLM裁判过滤无效任务；重写器将冗长的程序式指令压缩为自然、意图水平的用户请求，再经裁判验证。

#### 阶段2：轨迹合成（Trajectory Synthesis）
- **教师代理**：GLM-5.1-FP8逐步调用API，并调用任务完成API结束。
- **LLM API模拟器**：为每个API调用生成结构化JSON响应，条件包括：
  - API规范、调用参数、任务描述、同app内历史调用-响应对、虚拟用户信息、当前日期时间。
- **模拟器规则**：数据一致性（跨调用保持状态）、数据多样性（混合相关/不相关结果）、数据真实性（符合字段约束）、模式合规（严格遵循输出模式）。
- **质量检查**：程序化验证参数、模式匹配、LLM裁判检查语义一致性，失败时重试（有限次数）。

#### 阶段3：轨迹过滤（Trajectory Filtering）
- 使用Gemini-3.1-Pro作为LLM裁判，多次判定轨迹是否正确，仅保留多数正确票的轨迹。

## 3. 实验设计
### 数据集与benchmark
- **AppWorld**：9个日常app，457个API，平均任务需10个API、45次调用。两个测试集：Test-N（168任务）、Test-C（417任务，含两个未见app）。
- **OfficeBench**：8个app，20个API，任务需跨app长程规划。评估26个两app任务和46个三app任务（排除OCR依赖和不可解任务）。

### 对比方法
- **零样本基线**：多个现成模型（Gemini-3.1-Pro、GLM-5.1-FP8、GPT-4o、Nemotron-3-120B等）。
- **现有合成数据**：ToolAlpaca、ToolACE、Nemotron（简单工具调用，无状态）。
- **真实环境数据**：AppWorld训练集（AWT）：在真实环境中用GLM-5.1-FP8生成并过滤的634条轨迹。
- **ESAT数据变体**：
  - ESAT-S52：用52个合成app（1017个API，独立于AppWorld）生成6K轨迹。
  - ESAT-AW7：用AppWorld中7个app（排除测试集保留的2个）生成9K轨迹。
  - ESAT-OB：用OfficeBench 20个API生成1.7K轨迹。

### 评估指标
- AppWorld：任务目标完成率（TGC）和场景目标完成率（SGC）。
- OfficeBench：通过率（pass rate），按两app/三app/总体报告。
- 所有结果均为8次推理平均，附标准差。

### 额外分析
- 裁判模型消融（GLM vs Gemini过滤效果）。
- 模拟器质量分析：用GPT-5.1评估27K调用，模拟器有效响应率93.7%。
- 模拟器失败率随输出长度变化（短响应2-3%，>2K tokens时达23%）。
- 与人类标注一致性：95%一致率，Cohen's kappa=0.90。

## 4. 资源与算力
- **数据合成**：使用GLM-4.7-FP8（任务生成/裁判）、GLM-5.1-FP8（教师代理和模拟器）、Gemini-3.1-Pro（轨迹过滤）。未明确给出总合成计算量（如GPU小时数）。
- **模型微调**：使用AdamW优化器，lr=2e-5，线性衰减，1% warmup。最大上下文长度32768 tokens，每设备batch size=1。
  - 小于14B参数模型：8×H100 GPU。
  - 14B及以上模型：8×B200 GPU。

- **推理评估**：使用vLLM，采样温度1.0，每个任务生成8个样本。

## 6. 主要结论与发现
1. **ESAT数据显著提升性能**：微调后模型在AppWorld上性能提升最高50.5%（Qwen3.5-9B Test-N），在OfficeBench上最高60.5%（Qwen3.5-2B两app任务）。
2. **超越真实环境数据**：在AppWorld上，使用ESAT-S52+AW7训练的模型在几乎所有设置下优于使用真实环境中收集的AWT数据训练的模型（提升达15.3%）。
3. **较小模型匹敌大型模型**：4B-8B参数模型微调后可超过GPT-4o和Nemotron-3-120B；Qwen3.5-27B接近教师模型GLM-5.1-FP8。
4. **模拟器有效性高**：93.7%的模拟响应被GPT-5.1判定为有效；人类评估与LLM裁判高度一致（95%）。
5. **现有合成数据不足**：ToolAlpaca、ToolACE、Nemotron等简单工具调用数据在复杂状态化任务上基本无效。
6. **裁判模型选择关键**：Gemini作为轨迹裁判显著优于GLM，通过严格过滤提升下游性能。

## 7. 优点
- **创新性**：首次提出完全无需可执行环境的API代理轨迹合成框架，将数据生成简化为仅需API规范的规格问题，极大降低扩展成本。
- **方法设计**：
  - 分桶生成+逆频率采样确保任务多样性和API覆盖（AppWorld 340 API中99.7%被覆盖）。
  - 状态化模拟器维护跨调用一致性，支持写操作和复杂多步交互。
  - 多重质量检查（程序化验证+LLM裁判）保证数据准确性。
- **实验全面性**：覆盖多个模型尺度、多个benchmark、多种数据配置、消融实验、人工验证，结果扎实。
- **实用性**：合成的数据可直接用于SFT，效果显著，且能泛化到未见app。

## 8. 不足与局限
- **模拟器局限性**：模拟器失败率随输出长度增加（>2K tokens时达23%），可能限制长轨迹生成；仅依赖GLM-5.1-FP8，更强大的模型可能提升模拟质量。
- **计算成本**：虽无需真实环境，但合成过程仍需要大量LLM调用（尤其是模拟器和裁判），对资源要求不低。
- **泛化性有限**：仅在AppWorld和OfficeBench两个benchmark上验证，它们均属于特定领域（日常app和办公自动化），对其他类型API（如云服务、数据库）是否适用未知。
- **任务分布偏差**：生成的任务偏向单app（62-83%），多app任务较少，可能影响多app协作能力训练。
- **潜在偏差**：依赖LLM作为裁判和生成器，可能引入LLM自身的偏见或错误；合成数据分布可能与真实世界数据分布存在差距。
- **未讨论部署影响**：论文未讨论使用合成数据训练的模型在真实生产环境中的表现和安全性（如错误API调用的后果）。

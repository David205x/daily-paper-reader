## 摘要
大型语言模型（LLM）在将自然语言翻译为SQL方面展现了卓越能力，但现有方法在处理需要多步、数据感知推理的复杂查询时仍显不足。我们提出DecoSearch，一种无需训练的框架，通过将每个查询路由到适当级别的推理努力来解决这一问题。轻量级模式选择器首先将完整数据库模式修剪为相关的表和列。LLM判断器随后决定问题是否需要分解：直接的问题走直接生成路径，复杂的问题则升级到原子子问题的有向无环图（DAG），每个子问题由针对性的SQL生成步骤解决。RAG组件利用语义相似的训练示例为分解器提供依据，拓扑精炼器在执行失败表明分解有缺陷（而非可修复的SQL错误）时重构推理计划。DecoSearch在BIRD上达到70.53%的执行准确率，在Spider上达到88.31%（使用DeepSeek骨干网络），超越了所有无需训练的基线方法，同时消耗的令牌数比竞争方法少一个数量级。它还可以作为模型无关的包装器，无需修改流程即可持续改进微调后的SQL生成骨干网络。

## Abstract
Large Language Models (LLMs) have demonstrated remarkable capabilities in translating natural language to SQL, yet existing methods still falter on complex queries requiring multi-step, data-aware reasoning. We introduce DecoSearch, a training-free framework that addresses this by routing each query to the appropriate level of reasoning effort. A lightweight Schema Selector first prunes the full database schema to the relevant tables and columns. An LLM Judger then decides whether the question requires decomposition: straightforward questions follow a direct generation path and complex ones are escalated to a Directed Acyclic Graph (DAG) of atomic sub-questions, each solved by a targeted SQL generation step. A RAG component grounds the decomposer with semantically similar training examples, and a Topology Refiner restructures the reasoning plan when execution failures signal a flawed decomposition rather than a fixable SQL error. DecoSearch achieves 70.53% execution accuracy on BIRD and 88.31% on Spider with a DeepSeek backbone, surpassing all training-free baselines while consuming an order of magnitude fewer tokens than competing methods. It also functions as a model-agnostic wrapper, consistently improving fine-tuned SQL generation backbones without any modification to the pipeline.

- **研究动机**：需要一种能灵活分配推理资源的训练无关框架，根据查询复杂度动态选择直接生成或结构化分解，并能在计划层面修复错误的分解，从而提升复杂查询的准确率，同时保持高效率。
- **整体含义**：提出 DecoSearch，通过复杂度感知路由（Judger-led escalation）和计划级修复（Topology Refinement）来组织 Text-to-SQL 生成，在无需训练的情况下显著超越现有训练无关基线，并且可作为模型无关包装器提升微调模型性能。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 2.1 整体流程
- **Pipeline**：预处理（Schema Selector 修剪模式）→ Judger 判断复杂度（直接路径或分解路径）→ 若简单则直接生成 SQL 并执行；若复杂则进入分解路径：Decomposer 生成 DAG 原子子问题 → 按拓扑顺序逐个解决子问题 → 若某个节点执行失败则触发 Topology Refiner 重构 DAG 并重试。

### 2.2 核心组件
- **Schema Selector**：两阶段 LLM 调用：先提取关键词和实体，再基于关键词选择相关表和列。减少上下文噪声并降低 token 消耗。
- **Judger**：零样本 LLM 调用，输出 JSON 决策（`needs_decomposition: true/false`）及理由。优先走直接路径，仅在多粒度聚合、极端连接复杂度、多阶段推理依赖中间结果等情况下才分解。
- **Direct Path**：直接生成 SQL 并执行，成功则返回；失败则回退到分解路径。
- **Decomposer**：将复杂问题分解为 DAG 节点，每个节点是一个原子子问题，节点间用 `@[n]` 占位符表示依赖关系。利用 RAG 检索相似训练示例作为 few-shot 提示，提升分解质量。依赖注入：小结果集直接内联为值列表，大结果集物化为临时表。
- **SQL Generator**：每个节点一次零温度 LLM 调用生成 SQL，执行反馈用于触发拓扑精炼。
- **Topology Refiner**：当节点执行失败时，将当前 DAG 结构、失败节点和错误信息输入 LLM，要求生成修订后的 DAG（可重新划分依赖、插入中间节点、重新连线等）。限定最大重试次数 B，失败后继续下游节点实现优雅降级。

### 2.3 算法流程（Algorithm 1）
1. SchemaSelector 修剪模式。
2. Judger 判断：若选择直接路径，生成并执行 SQL；成功则返回；否则进入分解。
3. Decomposer 生成 DAG，拓扑排序得到队列。
4. 循环处理队列：子问题替换占位符 → 生成 SQL → 执行 → 若失败则调用 TopologyRefiner 更新 DAG 并重新排序队列；若成功则物化结果。
5. 返回最终根节点结果。

- DecoSearch 在 BIRD 上达到 70.53%，Spider 上 88.31% 执行准确率，**超越所有训练无关基线**，并在 Spider 上超过部分微调模型。

## 7. 优点

- **创新性**：首次系统性区分实现级、策略级、计划级的失败模式，并引入专门针对计划级缺陷的拓扑精炼，而非仅修复 SQL 语法。
- **高效性**：复杂度感知路由确保大多数简单查询以低成本直接生成，避免不必要的分解开销；RAG 组件帮助分解器产生更合理的 DAG。
- **实用性**：训练无关，仅需 API 调用；可作为即插即用包装器提升现有模型，无需重新训练。
